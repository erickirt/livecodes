## High-Level Architecture

LiveCodes is a client-side code playground that runs entirely in the browser. The architecture follows a modular design with clear separation of concerns.


## Core Systems

### 1. Entry Points

LiveCodes has multiple entry points depending on the use case:

| Entry Point | File                     | Purpose                                    |
| ----------- | ------------------------ | ------------------------------------------ |
| SDK         | `src/sdk/index.ts`       | JavaScript/TypeScript SDK for embedding    |
| App         | `src/livecodes/index.ts` | Main application entry with loading screen |
| Main        | `src/livecodes/main.ts`  | Orchestrates app loading in iframe         |

```mermaid
sequenceDiagram
    participant User
    participant SDK as createPlayground()
    participant Main as main.ts
    participant Index as index.ts
    participant Core as core.ts

    User->>SDK: createPlayground(container, config)
    SDK->>Main: Load app.html with config
    Main->>Main: Check loading mode (lazy/eager/click)
    Main->>Index: Initialize app
    Index->>Core: Initialize with config
    Core->>Core: Set up stores, events, compilers
    Core-->>Index: API object
    Index-->>Main:appLoaded event
    Main-->>SDK: ready event
    SDK-->>User: Playground object with API methods
```

### 2. Core Controller (`core.ts`)

The core controller is the orchestrator of the entire application. It manages:

- **State Management**: Configuration, editors, cache
- **Lifecycle**: Initialization, destruction, event handling
- **Coordination**: Connecting all subsystems

Key responsibilities:

```typescript
// Core state (module-level)
const stores: Stores = createStores();
const eventsManager = createEventsManager();
let editors: Editors;
let compiler: Compiler;
let formatter: Formatter;
let toolsPane: ToolsPane;
```

Core initialization flow:

```mermaid
flowchart TD
    A[app.ts called] --> B[Initialize Stores]
    B --> C[Initialize Events Manager]
    C --> D[Create Notifications]
    D --> E[Load i18n]
    E --> F[Create Editors]
    F --> G[Load Compilers]
    G --> H[Create Tools Pane]
    H --> I[Apply Initial Config]
    I --> J[Load Starter Templates]
    J --> K[Attach Event Handlers]
    K --> L[Run Initial Code]
    L --> M[Dispatch 'ready' Event]
```

### 3. Configuration System

The configuration system manages all application state:

```mermaid
flowchart LR
    subgraph Sources
        A[Default Config]

        E[Hash Params]
    end

    subgraph Processing
        F[buildConfig]
        G[upgradeConfig]
        H[validateConfig]
    end

    subgraph State
        I[appConfig<br/>runtime state]
    end

    A --> F
    B --> F
    C --> F
    D --> F
    E --> F
    F --> G
    G --> H
    H --> I
```

**Configuration Categories**:

| Category        | Description                               | Extractor              |
| --------------- | ----------------------------------------- | ---------------------- |
| ContentConfig   | Project content (markup, style, script)   | `getContentConfig()`   |
| AppConfig       | App behavior (readonly, mode, tools)      | `getAppConfig()`       |
| UserConfig      | User preferences (theme, autoupdate)      | `getUserConfig()`      |
| EditorConfig    | Editor settings (font, line numbers)      | `getEditorConfig()`    |
| FormatterConfig | Formatting settings (tabSize, semicolons) | `getFormatterConfig()` |

## Data Flow

### Compilation Flow

```mermaid
sequenceDiagram
    participant User
    participant Editor
    participant Core
    participant Compiler
    participant Worker
    participant Result

    User->>Editor: Type code
    Editor->>Core: onContentChanged
    Core->>Core: updateConfig()
    Core->>Core: Check autoupdate

    alt autoupdate enabled
        Core->>Compiler: compile(markup, language, config)
        Compiler->>Worker: postMessage(compile)
        Worker->>Worker: Load language compiler
        Worker->>Worker: Execute compilation
        Worker-->>Compiler: postMessage(compiled)
        Compiler->>Core: CompileResult
        Core->>Result: getResultPage()
        Result->>Result: Create HTML
        Result-->>Core: HTML string
        Core->>Core: createIframe()
        Core->>Result: postMessage(result)
    end
```

### Result Page Generation

The result page is generated by combining compiled code from all editors:

```mermaid
flowchart TD
    A[getResultPage] --> B[Compile Markup]
    B --> C[Compile Style]
    C --> D[Compile Script]
    D --> E[Compile Tests]

    E --> F{Processors?}
    F -->|Yes| G[Apply PostCSS/Tailwind/etc]
    F -->|No| H[Create Import Map]
    G --> H

    H --> I[Create Result HTML]
    I --> J[Add CS Preset]
    J --> K[Add External Resources]
    K --> L[Add Runtime Scripts]

    L --> M{Script Type}
    M -->|module| N[Add as ES Module]
    M -->|classic| O[Add as Classic Script]

    N --> P{Live Reload?}
    O --> P
    P -->|Supported| Q[postMessage to iframe]
    P -->|Not supported| R[Reload iframe]

    Q --> S[Update Cache]
    R --> S
```

```mermaid
    }

        [*] --> InitializeCore
    }
```

### Run Code Workflow

When the user triggers code execution (Run button or auto-update):

```mermaid
flowchart TD
    A[run triggered] --> B[updateConfig]
    B --> C{Style only update?}

    C -->|Yes| D[Post CSS update to iframe]
    C -->|No| E[Get all compiled code]

    D --> F[Update Cache]

    E --> G[compile markup]
    G --> H[compile style<br/>with HTML context]
    H --> I[compile script]
    I --> J{Tests enabled?}

    J -->|Yes| K[compile tests]
    J -->|No| L[createResultPage]
    K --> L

    L --> M[Apply processors]
    M --> N[Create import map]
    N --> O[Generate HTML]

    O --> P{LiveReload support?}
    P -->|Yes| Q[postMessage to iframe]
    P -->|No| R[Reload iframe]

    Q --> F
    R --> F

    F --> S[Update compiled code viewer]
    S --> T[Clear console if needed]
    T --> U[Dispatch change event]
```

## System Interactions

### Event System

LiveCodes uses a publish/subscribe pattern for cross-component communication:

```mermaid
flowchart LR
    subgraph Publishers
        Core[Core Controller]
        SDK[SDK Watchers]
        Iframe[Result Iframe]
    end

    subgraph EventBus
        Events[Events Manager<br/>addEventListener/removeEventListener]
        CustomEvents[Custom Events<br/>livecodes-*]
        PubSub[Pub/Sub<br/>createPub]
    end

    subgraph Subscribers
        UI[UI Components]
        ToolsPane[Tools Pane]
        Storage[Storage Sync]
        External[External Handlers]
    end

    Core --> Events
    SDK --> CustomEvents
    Iframe --> CustomEvents

    Events --> UI
    CustomEvents --> ToolsPane
    Events --> Storage
    CustomEvents --> External
```

**Custom Events**:

| Event                     | When Fired           | Payload              |
| ------------------------- | -------------------- | -------------------- |
| `livecodes-init`          | App starts loading   | `{ appVersion }`     |
| `livecodes-ready`         | Project loaded       | -                    |
| `livecodes-load`          | Load triggered (SDK) | -                    |
| `livecodes-change`        | Code changed         | `{ code, config }`   |
| `livecodes-test-results`  | Tests completed      | `{ results, error }` |
| `livecodes-console`       | Console output       | `{ method, args }`   |
| `livecodes-destroy`       | App destroyed        | -                    |
| `livecodes-resize-editor` | Editor resize        | -                    |

**Pub/Sub for SDK**:

```typescript
const sdkWatchers = {
  load: createPub<void>(),
  ready: createPub<void>(),
  code: createPub<{ code: Code; config: Config }>(),
  tests: createPub<{ results: TestResult[]; error?: string }>(),
  console: createPub<{ method: string; args: any[] }>(),
  destroy: createPub<void>(),
};
```

### Compiler System

The compiler system uses Web Workers for performance:

```mermaid
flowchart TB
    subgraph Main Thread
        Core[core.ts]
        CompilerFactory[getCompiler]
        CreateCompiler[createCompiler]
    end

    subgraph Worker Module
        Worker[compile.worker.ts]
        CompilerRegistry[getAllCompilers]
        LanguageLoaders[Language Compilers]
    end

    subgraph Page Context
        Sandbox[compiler-sandbox.ts<br/>iframe]
        PageCompilers[compile.page.ts]
    end

    Core --> CompilerFactory
    CompilerFactory --> CreateCompiler
    CreateCompiler -->|Worker supported| Worker
    CreateCompiler -->|Not supported| Sandbox

    Worker --> CompilerRegistry
    Worker --> LanguageLoaders

    Sandbox --> PageCompilers

    subgraph Message Protocol
        M1[init]
        M2[load]
        M3[compile]
        M4[compiled]
        M5[error]
    end

    CreateCompiler<-->|postMessage| Worker
    CreateCompiler<-->|postMessage| Sandbox
```

**Worker vs Page Context**:

| Compiler Type           | Environment  | Reason                      |
| ----------------------- | ------------ | --------------------------- |
| Most languages          | Worker       | Performance, no UI blocking |
| ReScript, OCaml, Reason | Page/Sandbox | Needs DOM APIs              |
| MDX, Diagrams           | Page/Sandbox | WebAssembly limitations     |
| PostgreSQL              | Page/Sandbox | Creates nested workers      |
